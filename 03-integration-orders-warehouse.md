# Интеграция: интернет-магазин → «1С:Управление торговлей 11»

**Роль:** бизнес-аналитик (проектная документация)  
**Источник:** веб-витрина на **1С-Битрикс: Управление сайтом** (модуль интернет-магазина), REST API обмена заказами  
**Получатель:** **1С:Управление торговлей**, редакция 11.5, типовая конфигурация с документом «Заказ клиента»  
**Контур:** облако AWS EU (фронт) + on-premise 1С на площадке заказчика в Москве, обмен по HTTPS через шину **RabbitMQ 3.12**  
**Дата документа:** 29.04.2026  
**Владелец интеграции со стороны бизнеса:** операционный директор ООО «ТехноСнаб» (учебное название)

---

## 1. Цель и контекст

**Бизнес-цель:** автоматически создавать в 1С документ основания для отгрузки после оплаты заказа на сайте, убрать ручной ввод оператором.

**As-Is:** оператор 1–2 раза в час выгружает оплаченные заказы из админки Битрикс в Excel и ручками заводит «Заказ клиента». Средняя задержка от оплаты до появления документа в 1С — **47 минут**. Ошибки коэффициента пересчёта количества (короб vs штука) — **около 1,1%** строк.

**To-Be:** при переходе заказа в статус «Оплачен» в Битрикс в шину публикуется событие `order.paid`; сервис-адаптер вызывает HTTP-сервис 1С и создаёт документ в статусе «К выполнению». Целевая задержка p95 — **3 минуты**.

---

## 2. Триггеры и события обмена

| Событие в Битрикс | Топик RabbitMQ | Отправлять в 1С? | Поведение в 1С |
|-------------------|----------------|------------------|----------------|
| `ORDER_PAID` (оплачен полностью) | `bitrix.order.paid` | Да | Создать или обновить «Заказ клиента» |
| `ORDER_CANCELLED` до отгрузки | `bitrix.order.cancelled` | Да | Установить пометку отмены / закрыть заказ (политика заказчика L1) |
| `ORDER_PARTIALLY_SHIPPED` | не в MVP | Нет | Фаза 2 через документ «Реализация» |
| Изменение только комментария клиента | — | Нет | Не поднимаем нагрузку |

**Политика L1 при отмене:** если в 1С уже есть отгрузка — событие отмены уходит в очередь ручного разбора (DLQ `1c.manual.review`).

## 3. Маппинг полей

| Поле JSON (источник) | Реквизит 1С «Заказ клиента» | Правило преобразования |
|----------------------|----------------------------|-------------------------|
| `order_id` | `НомерВнешнегоДокумента` | Строка, префикс `WEB-` |
| `occurred_at` | `Дата` | UTC → `Europe/Moscow` (ООО зарегистрировано в МСК) |
| `customer.external_id` | `Контрагент` (справочник) | Поиск по `IDКонтрагентаВБитрикс`; при отсутствии — создать элемент по шаблону «Розница физлицо» |
| `customer.phone` | `КонтактныйТелефон` | Нормализация к `+7XXXXXXXXXX`, regex валидатора адаптера |
| `customer.full_name` | `КонтактноеЛицо` | Строка ≤ 150 символов, trim |
| `delivery.address_line` | `АдресДоставки` | Конкатенация: индекс, город, улица, дом, кв |
| `lines[].sku` | `Номенклатура.Код` | Таблица соответствия `bitrix_product_id ↔ код 1С`; ошибка при отсутствии |
| `lines[].qty` | `Количество` | Целое > 0; для дробных упаковок — отдельная строка `unit` → коэффициент из справочника `ЕдиницыИзмерения` |
| `lines[].price` | `Цена` | Передаётся как справка; в 1С тип цены «Розничная сайт»; факт пересчёта в типовой логике |
| `payment.transaction_id` | `НомерТранзакцииЭквайринга` | Для сверок с банком |
| `payment.amount_rub` | `СуммаДокумента` | Контроль ±0,01 ₽ с полем `lines sum` |

**Владелец справочника соответствия SKU:** отдел каталога + главный бухгалтер (подпись протокола раз в квартал).

---

## 4. Последовательность вызовов (sequence)

1. **Битрикс:** статус заказа `P` (оплачен), модуль интеграции формирует тело JSON и публикует в `bitrix.order.paid`.  
2. **Адаптер (microservice `bitrix-1c-bridge`):** потребляет сообщение, проверяет подпись HMAC-SHA256.  
3. **Адаптер → 1С:** `POST https://1c.technosnab.local/hs/order/import` с Basic Auth, body JSON, заголовок `X-Idempotency-Key: {order_id}`.  
4. **1С:** валидация, поиск контрагента, заполнение табличной части. Успех → HTTP `200`, тело `{ "doc_number": "ЗК-00012345", "doc_uuid": "…" }`.  
5. **Адаптер:** пишет ответ в Битрикс (служебное свойство заказа `UF_1C_DOC`), ACK в RabbitMQ.  
6. **Ошибка 4xx от 1С:** не ACK, retry с экспоненциальной задержкой **2 мин**, максимум **5** попыток, затем DLQ + алерт в Telegram интегратору.

---

## 5. Обработка ошибок и идемпотентность

| Ситуация | Поведение |
|----------|-----------|
| Повторная доставка того же `order_id` | 1С не создаёт второй документ; возвращает `200` с тем же `doc_uuid` |
| Нет ответа 1С 60 с | Таймаут адаптера, повтор по политике §4 п.6 |
| Неизвестный `sku` | HTTP `422` от адаптера до вызова 1С; заказ в Битрикс помечается `UF_INTEGRATION_ERROR=SKU`; письмо катологу |
| Расхождение суммы > 1 ₽ | Блокировать создание до ручного подтверждения в панели адаптера |
| RabbitMQ недоступен | Битрикс пишет в локальный `b_integration_queue` (MySQL), воркер догоняет |

---

## 6. Нефункциональные требования

- Задержка «оплачен на сайте» → «документ проведён в 1С»: **p95 ≤ 3 мин**, **p99 ≤ 8 мин** в часы 09:00–21:00 МСК.  
- Логирование: полный JSON запроса/ответа с **маскировкой** номера карты (не храним PAN).  
- Мониторинг: Grafana — метрики `queue_lag`, `bridge_errors`, `1c_latency_ms`.  
- Версионирование контракта JSON: поле `schema_version: "1.0"`.

## 7. Тест-кейсы

| ID | Сценарий | Вход | Ожидаемый результат |
|----|----------|------|---------------------|
| TC-01 | Один SKU, оплата | Заказ 1 строка, `payment.status=paid` | В 1С один «Заказ клиента», количество = входному |
| TC-02 | Три строки | 3 разных SKU | Три строки ТЧ, сумма = 12 470,00 ₽ |
| TC-03 | Отмена до отгрузки | `ORDER_CANCELLED` | Документ помечается согласно L1, слот товара освобождён |
| TC-04 | Неизвестный SKU | `sku=UNKNOWN-999` | Ошибка SKU, документ не создан, флаг в Битрикс |
| TC-05 | Дубль webhook | Два одинаковых сообщения подряд | Один документ в 1С, дважды `200` с одним `doc_uuid` |
| TC-06 | Заказ не оплачен | `status=awaiting_payment` | Событие не публикуется / игнорируется адаптером |
| TC-07 | Таймаут 1С | Mock 60 с sleep | 5 ретраев, затем DLQ, алерт |
| TC-08 | Пустой адрес | `address_line=""` | `422`, запись в лог, заказ не уходит в 1С |
| TC-09 | qty = 9999 | Лимит заказа 500 ед. на SKU | `422` из адаптера по бизнес-правилу крупного заказа |
| TC-10 | Unicode в ФИО | `Иванова-Петрова Анна 玛利亚` | Корректная запись в 1С, проверка отображения в печатной форме |
| TC-11 | Частичная скидка в корзине | `discount_total=350` | В 1С отдельная строка скидки или пересчёт по правилу заказчика — как в спецификации №2 |
| TC-12 | Контрагент новый | phone не найден | Создание контрагента-розницы, ИНН пустой |
| TC-13 | Неверный HMAC | Подпись битая | HTTP `401` адаптера, сообщение не ACK |
| TC-14 | Версия схемы 0.9 | `schema_version` устарела | `426 Upgrade` от адаптера, инструкция обновить модуль |
| TC-15 | Смешанные размерности | 2 шт + 1,5 кг (два разных `unit`) | Две строки ТЧ с корректными единицами из справочника |

## 8. Вопросы, закрытые на проектном комитете 15.04.2026

- **Скидки:** строкой «Скидка по промокоду» с текстом промокода, без проведения в отдельном документе.  
- **Частичная отгрузка:** перенесена на фазу 2 (отдельный BRD).
