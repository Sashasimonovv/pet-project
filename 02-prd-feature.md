# PRD: фича «Повторить последний заказ» в приложении FoodFast

**Продукт:** FoodFast — агрегатор доставки готовой еды и продуктов (учебный продукт для портфолио; не связан с реальным брендом)  
**Платформы:** iOS 16+, Android 11+  
**Версия PRD:** 1.0  
**Автор:** бизнес-аналитик (портфолио-кейс)  
**Дата:** 29.04.2026  
**Согласование:** продакт-оунер: условное лицо; tech lead: условное лицо

---

## 1. Контекст и проблема

**JTBD:** Когда я несколько раз заказываю один и тот же набор блюд в одну и ту же квартиру, я хочу оформить заказ **за минимум шагов**, чтобы не тратить время на повторный выбор корзины и адреса.

**Текущая ситуация (по внутренней аналитике учебного кейса):** 34% активных пользователей за последние 30 дней сделали **≥2 успешных заказа**. Из них 61% каждый раз проходят полный путь «ресторан → корзина → чекаут», хотя в 47% повторных заказах состав отличается **не более чем на одну позицию** от предыдущего успешного заказа.

**Гипотеза:** CTA «Повторить заказ» на главном экране для сегмента «есть успешный заказ за последние 90 дней» повысит долю повторных заказов и снизит медианное время от входа в приложение до экрана успешной оплаты в этом сегменте.

## 3. Аудитория и сегменты

| Сегмент | Описание | Показываем CTA? |
|---------|----------|-----------------|
| R1 — активные повторяющие | ≥1 успешный заказ со статусом `delivered` за 90 дней, пользователь авторизован | Да |
| R2 — гости | Заказ без регистрации или сессия без аккаунта | Нет |
| R3 — только отмены / возвраты | Нет ни одного `delivered` за 90 дней | Нет |
| R4 — юридический стоп | Пользователь из региона с ограничением на сохранение истории (настройки приватности) | Нет |

**Развёртка:** 10% пользователей в сегменте R1 — **A/B тест**: контроль без CTA, вариант с CTA (фича-флаг `repeat_order_v1`).

## 4. Объём работ (Scope)

**Must (MVP релиз 1.0)**

- Карточка на главной: «Повторить заказ от 27 апреля» (дата последнего `delivered`).  
- По тапу — экран **проверки**: состав строк заказа с актуальной ценой, ресторан, адрес доставки, чаевые по умолчанию, способ оплаты по умолчанию.  
- Кнопка «Оформить и оплатить» ведёт в существующий платёжный шлюз.  
- При успешной оплате создаётся новый `order_id`; история заказов хранится неизменной.

**Should (релиз 1.1, через спринт)**

- Если позиция недоступна — модальное окно: «Убрать из заказа» / «Заменить на похожее» (рекомендация из категории).  
- Если ресторан закрыт или не доставляет по адресу — показать это до оплаты... + ссылка «Выбрать другой ресторан».

**Could**

- Горизонтальный список из **трёх** последних успешных заказов для выбора.

**Out of scope**

- Подписки «автозаказ раз в неделю».  
- Объединение корзины «повтор + новое».  
- Отдельные промокоды только за использование «Повтор».

## 5. User flow

**Основной поток**

1. Пользователь авторизован, открывает приложение, геопозиция и адрес по умолчанию актуальны.  
2. На главной в блоке «Снова заказать» отображается карточка с миниатюрой 3 блюд, суммой «от … ₽», рестораном.  
3. Тап → экран `RepeatOrderReview` с полным списком позиций, итогом, временем доставки по умолчанию.  
4. Пользователь может изменить адрес / время / чаевые (существующие компоненты).  
5. Тап «Оформить за N ₽» → 3-D Secure при необходимости → экран `OrderConfirmed`.

**Альтернативные потоки**

- **A1:** Нет сети — показ `OfflineSnackbar`, кнопка не активна.  
- **A2:** Цена изменилась более чем на 10% от суммы прошлого заказа — показ баннера «Итог изменился из-за цен ресторана» с обязательным скроллом до конца баннера перед оплатой.  
- **A3:** Адрес удалён — редирект на `AddressPicker`, после выбора возврат на `RepeatOrderReview`.

## 6. Функциональные требования

| ID | Требование |
|----|------------|
| FR-1 | «Последний успешный заказ» = максимальный по `delivered_at` среди заказов пользователя со статусом `delivered`, не помеченных фродом. |
| FR-2 | CTA показывается только если `feature_flag.repeat_order_v1 = true` и сегмент R1. |
| FR-3 | При сборке корзины цены подгружаются из **текущего меню** ресторана; если SKU не существует, позиция помечается `unavailable`. |
| FR-4 | Нельзя перейти к оплате, пока в заказе есть хотя бы одна `unavailable` без действия пользователя (удаление/замена). |
| FR-5 | Способ оплаты по умолчанию = последний успешно использованный в приложении; если карта протухла — выбор новой. |
| FR-6 | Событие `repeat_order_shown` содержит `source_screen=home`, `restaurant_id`, `baseline_order_id`. |

## 7. Нефункциональные требования

| ID | Требование |
|----|------------|
| NFR-1 | p95 ответа API `GET /orders/last-delivered` ≤ 350 мс при нагрузке до 500 RPS (горизонтальное масштабирование сервиса заказов). |
| NFR-2 | Локализация RU/EN; строки в Lokalise-ключах `repeat_order.*`. |
| NFR-3 | Логи ошибок с `correlation_id` при падении оплаты после старта сценария повтора. |
| NFR-4 | Доступность экрана для TalkBack / VoiceOver: заголовок, список позиций, итог озвучиваются. |

**Аналитика (события)**

- `repeat_order_shown`, `repeat_order_tap`, `repeat_order_review_open`, `repeat_order_pay_start`, `repeat_order_success`, `repeat_order_fail` (с `reason_code`: `payment_declined`, `restaurant_closed`, `validation_error`, …).

## 8. Критерии приёмки (общие)

- [ ] Пользователь R2 никогда не видит CTA «Повтор».  
- [ ] После `repeat_order_success` в БД существует новый заказ с корректным `restaurant_id`, суммой и статусом `paid`.  
- [ ] При недоступной позиции оплата технически невозможна до явного действия пользователя.  
- [ ] Все события из списка в разделе 7 реально уходят в хранилище событий в 99,9% случаев (мониторинг за 7 дней пилота).
